# 图片生成超时优化方案

## 背景

当前图片生成请求在遇到 `curl: (92) HTTP/2 stream` 等网络错误时，会因多重超时级联导致单次请求耗时 1777 秒（约 30 分钟）才失败，用户体验极差。

### 根因

1. **curl: (92) 未被快速识别**：`is_tls_connection_error` 和 `is_connection_timeout_error` 均不识别 HTTP/2 流错误，导致它在 `_poll_image_results` 内部被当作普通 `RequestException`，通过 `_retry_sleep` 持续重试直到 300s 超时预算耗尽
2. **轮询超时被强制提升**：[conversation.py:994](file:///d:/GithubProject/chatgpt2api/services/protocol/conversation.py#L994) 和 [1098](file:///d:/GithubProject/chatgpt2api/services/protocol/conversation.py#L1098) 的 `max(config.image_poll_timeout_secs, 300)` 使 config.json 中配置的 120s 失效
3. **外层 4 次换账号重试无总耗时上限**：[conversation.py:1251](file:///d:/GithubProject/chatgpt2api/services/protocol/conversation.py#L1251) `MAX_POLL_TIMEOUT_RETRIES = 4`，每次重试都重新走完整流程（含 300s 轮询），累计 1200s+
4. **`image_timeout_retry_secs` 配置缺失**：config.json 和前端都有此字段（值=30），但 [services/config.py](file:///d:/GithubProject/chatgpt2api/services/config.py) 无对应 property，后端读不到

### 耗时累积公式

```
curl: (92) → _poll_image_results 内部重试到 300s 耗尽 → ImagePollTimeoutError
→ 外层换账号重试 4 次 × 300s = 1200s
+ SSE/bootstrap/upload 等开销 ≈ 1777s
```

## 目标

- 引入全局总耗时上限（默认 300s，可在 config.json 配置），超过立即失败
- curl: (92) 等 HTTP/2 流错误快速重试 2 次后立即抛出，不耗尽轮询预算
- 尊重 config.json 中 `image_poll_timeout_secs` 配置，移除强制 300s 下限
- 补充 `image_timeout_retry_secs` 后端 property，让配置生效

### 改动 1：新增 `image_total_timeout_secs` 和 `image_timeout_retry_secs` 配置项

**文件**：`d:\GithubProject\chatgpt2api\services\config.py`
**位置**：在 [line 453](file:///d:/GithubProject/chatgpt2api/services/config.py#L453) `image_settle_secs` property 之后

**新增**：
```python
@property
def image_total_timeout_secs(self) -> int:
    """图片生成请求的总耗时上限（秒）。超过此时间立即失败，不再重试。
    包含所有重试和轮询时间，是硬性上限。"""

    """单个账号轮询超时后，续轮询的额外等待时间（秒）。"""
    return max(1, int(self.data.get("image_timeout_retry_secs", 30)))
```

**原因**：
- `image_total_timeout_secs` 是本次优化的核心配置，控制全局 deadline
- `image_timeout_retry_secs` 在 config.json（line 97）和前端已存在，但后端从未读取，属于存量 bug

### 改动 2：新增 HTTP/2 流错误识别函数

**文件**：`d:\GithubProject\chatgpt2api\services\protocol\conversation.py`
**位置**：在 [line 103](file:///d:/GithubProject/chatgpt2api/services/protocol/conversation.py#L103) `is_connection_timeout_error` 函数之后

**新增**：
```python
def is_http2_stream_error(message: str) -> bool:
    """检测 HTTP/2 流错误和网络层致命错误。
    这类错误（如 curl 92/56/95）表示上游流被重置或连接异常断开，
    不应长时间重试，应快速失败后由外层决定是否换账号。"""
    text = str(message or "").lower()
    return (
        "curl: (92)" in text          # HTTP/2 stream not closed cleanly
        or "curl: (56)" in text        # Failure with receiving network data
        or "curl: (95)" in text        # HTTP/3 stream error
        or "http/2 stream" in text
        or "http/2 protocol error" in text
        or "stream error" in text
        or "rst_stream" in text
        or "internal_error" in text    # curl: (92) 的 INTERNAL_ERROR 子串
    )
```

**原因**：当前没有任何函数识别 HTTP/2 流错误，导致 curl: (92) 被当作普通网络错误在 `_poll_image_results` 内部耗尽 300s 预算。

### 改动 3：`_poll_image_results` 内部对 HTTP/2 流错误快速重试

**文件**：`d:\GithubProject\chatgpt2api\services\openai_backend_api.py`
**位置**：[line 2084-2273](file:///d:/GithubProject/chatgpt2api/services/openai_backend_api.py#L2084-L2273) `_poll_image_results` 方法

**改动 3a：在方法内部新增网络错误计数器**
在 [line 2110](file:///d:/GithubProject/chatgpt2api/services/openai_backend_api.py#L2110) `has_initial_ids` 定义之后，新增：
```python
network_error_count = 0
MAX_NETWORK_ERRORS = 2  # HTTP/2 流错误等致命网络错误最多重试 2 次
```

**改动 3b：修改 `except RequestException` 分支**
将 [line 2196-2199](file:///d:/GithubProject/chatgpt2api/services/openai_backend_api.py#L2196-L2199)：
```python
except requests.exceptions.RequestException as exc:
    if _retry_sleep("network", None, str(exc), None):
        continue
    break
```
改为：
```python
except requests.exceptions.RequestException as exc:
    error_str = str(exc)
    # HTTP/2 流错误等致命网络错误：快速重试 2 次后立即抛出，不耗尽 timeout_secs
    if is_http2_stream_error(error_str) or is_tls_connection_error(error_str):
        network_error_count += 1
        if network_error_count <= MAX_NETWORK_ERRORS:
            # 短退避：2s, 4s（不使用 _retry_sleep 的指数退避，避免拖长）
            short_backoff = min(2.0 * network_error_count, max(0.0, _remaining()))
            logger.warning({
                "event": "image_poll_network_error_fast_retry",
                "conversation_id": conversation_id,
                "attempt": attempt,
                "network_error_count": network_error_count,
                "sleep_secs": round(short_backoff, 2),
                "error": error_str[:200],
            })
            if short_backoff > 0:
                time.sleep(short_backoff)
                continue
        # 重试次数用尽，立即抛出（不耗尽 timeout_secs）
        logger.warning({
            "event": "image_poll_network_error_exhausted",
            "conversation_id": conversation_id,
            "attempt": attempt,
            "network_error_count": network_error_count,
            "error": error_str[:200],
        })
        raise ImagePollTimeoutError(
            f"网络错误重试 {MAX_NETWORK_ERRORS} 次后仍失败: {error_str[:200]}"
        )
    # 其他网络错误：保持原有 _retry_sleep 逻辑
    if _retry_sleep("network", None, str(exc), None):
        continue
    break
```

**原因**：这是 1777s 的核心放大点。curl: (92) 原本会在内部重试到 300s 耗尽，现在改为快速重试 2 次（共约 6s）后立即抛出，让外层决定是否换账号。

**注意**：需要在文件顶部确认 `is_http2_stream_error` 和 `is_tls_connection_error` 的导入。当前 `is_tls_connection_error` 已在 conversation.py 定义，需确认 openai_backend_api.py 是否已导入，若未导入需添加 import。

    # 检查全局 deadline，剩余时间不足以再试一次则直接失败

## 不改动的部分（避免过度工程化）

1. **SSE 流读取超时**：`_stream_picture_conversation` 中 `iter_sse_payloads(response)` 无超时，但 `_start_image_generation` 的 `timeout=300` 已覆盖流建立阶段。当前案例中 SSE 流不是主要瓶颈，暂不改动。
2. **外层重试次数 `MAX_POLL_TIMEOUT_RETRIES = 4`**：保持不变，靠全局 deadline 自然限制。降低次数会减少成功机会，deadline 兜底更合理。
3. **`stream_image_outputs_with_pool` 并行场景**：并行多图时每个线程独立 deadline，暂不引入跨线程总 deadline（复杂度高，收益有限）。
4. **`ImageTaskService._run_task` 外层监督**：不增加额外监督线程，依赖底层 deadline 机制自然生效。

|---|---|---|
| curl: (92) 单次错误 | 耗尽 300s 轮询预算 | 快速重试 2 次（~6s）后抛出 |

## 风险评估

1. **误判失败风险**：全局 deadline 300s 可能导致部分慢请求被误判。缓解：用户可在 config.json 调大 `image_total_timeout_secs`。
2. **网络错误快速重试可能不够**：2 次重试对于持续抖动的网络可能不足。缓解：外层仍有 4 次换账号重试机会，每次都会重新尝试。
3. **存量配置兼容性**：`image_timeout_retry_secs` property 新增后，前端已存在的配置值（30）将首次在后端生效，需确认行为符合预期。
