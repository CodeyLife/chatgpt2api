# 提高 OpenAI 注册账号存活率方案

## 一、摘要

本方案针对 `chatgpt2api` 注册机系统，目标是**全链路提高账号存活率**：既提升注册过程成功率（降低 CF/sentinel/邮箱域名拦截），也提升注册后长期存活率（降低 refresh_token 失效、`app_session_terminated`、账号被 deactivate 的概率）。

方案定位为**中等改动**：聚焦反检测指纹体系的一致性、池化与持久化，以及注册节流，改动集中在 4 个核心文件，不重构整体架构。

## 二、现状分析

| 维度 | 实现位置 | 现状 |

| 请求头完整度 | `openai_register.py:61-108` | 完整 sec-ch-ua-* / sec-fetch-* / dnt / priority |
| Datadog Trace 伪造 | `openai_register.py:128-138` | 每请求随机 trace_id/parent_id |
| Sentinel PoW | `utils/sentinel.py` | FNV-1a 32 位暴力求解 |
| Cloudflare Clearance | `proxy_service.py` + FlareSolverr | 检测挑战→刷新 cookie+UA→重试一次 |
| PKCE | `utils/pkce.py` | 标准 RFC 7636 S256 |

| refresh_token 保活 | `account_service.py` `keepalive_refresh_tokens` | 每 3 天批量保活，失败退避 6h |

### 2.2 影响存活率的关键缺陷（本方案要解决的）

#### 缺陷 A：注册与重登的指纹严重不一致（高优先级）

注册流程与重登流程使用不同的指纹特征，会被 OpenAI 关联为"同账号跨设备/跨环境"，是 `app_session_terminated` 的重要诱因：

| 特征 | 注册流程 | 重登流程 | 位置 |
|---|---|---|---|
| `impersonate` | `"chrome"` | `"chrome110"` | `openai_register.py:259` vs `account_service.py:600` |
| `accept-language` | `en-US,en;q=0.9` | `zh-CN,zh;q=0.9,en;q=0.8` | `openai_register.py:64` vs `account_service.py:641` |
| `sec-ch-ua` | `'"Google Chrome";v="145"...'` | `'"Chromium";v="145", "Google Chrome";v="145", "Not/A)Brand";v="99"'` | `openai_register.py:53` vs `account_service.py:643` |
| `screen_hint` | `signup` | `login_or_signup` | 流程差异，可接受 |

#### 缺陷 B：device_id 不持久化（高优先级）

- 注册时 `PlatformRegistrar.__init__` 生成 `device_id = str(uuid.uuid4())`（`openai_register.py:386`），但 `worker()` 返回的 `result` dict（597-605 行）**只保存 email/password/access_token/refresh_token/id_token**，不保存 device_id。
- 重登时 `_login_with_password`（`account_service.py:607`）重新 `device_id = str(uuid.uuid4())`，与注册时的 device_id 完全不同。
- 后果：OpenAI 看到同一邮箱在不同 device_id 上登录，触发"设备指纹异常"风控，导致账号被 deactivate。

#### 缺陷 C：指纹特征全系统单一（中优先级）

- UA 固定 Chrome 145 / Windows 10（`openai_register.py:48-52`）。
- `sec_ch_ua` 固定（`openai_register.py:53-54`）。
- `sentinel.py:44` 屏幕分辨率固定 `"1920x1080"`，而 `pow.py` 已有 4 种分辨率随机——两套 PoW 体系环境特征不一致。
- 同 IP 批量注册时，所有账号共享同一套指纹，极易被聚类识别。

#### 缺陷 D：注册无间隔抖动（中优先级）

`register_service.py:274-303` 的 `_run()` 中，`executor.submit(openai_register.worker, submitted)` 在有空闲线程时立即提交，无线程间延迟。同 IP 短时间内并发注册易触发风控。

#### 缺陷 E：姓名池过小（中优先级）

`openai_register.py:157-160` 的 `_random_name()` 只有 8×7=56 种组合，注册量大时姓名重复率高，被识别为批量注册。

#### 缺陷 F：Sentinel SDK URL 硬编码日期版本（低优先级）

`sentinel.py:49` 硬编码 `"https://sentinel.openai.com/sentinel/20260124ceb8/sdk.js"`，含日期版本号 `20260124ceb8`，OpenAI 更新 SDK 后该字段可能失效，导致 sentinel token 被拒。

## 三、改进方案

### 改动 1：新增指纹 Profile 管理模块（核心基础设施）

**文件**：`d:\GithubProject\chatgpt2api\utils\fingerprint.py`（新建）

**目的**：统一管理注册/重登/刷新全流程的浏览器指纹，确保同一账号全生命周期指纹一致，不同账号指纹差异化。

**设计**：
```python
# 定义若干个完整的浏览器 Profile，每个 Profile 包含一致的特征组合
@dataclass
class BrowserProfile:
    impersonate: str          # curl_cffi 指纹标识，如 "chrome", "chrome110", "chrome120"
    user_agent: str
    sec_ch_ua: str
    sec_ch_ua_full_version_list: str
    sec_ch_ua_platform: str   # '"Windows"' / '"macOS"'
    sec_ch_ua_platform_version: str
    sec_ch_ua_arch: str
    sec_ch_ua_bitness: str
    accept_language: str      # 统一用 en-US,en;q=0.9（避免暴露真实语言）
    screen_resolution: str    # sentinel/pow 用，如 "1920x1080", "2560x1440"
    hardware_concurrency: int # 用于 sentinel config

# 预置 4-6 个 Profile，覆盖 Chrome 145/131/120 + Windows/macOS 组合
PROFILES: list[BrowserProfile] = [...]

def pick_profile(seed: str = "") -> BrowserProfile:
    """根据 seed（如 email）确定性选择 profile，保证同一账号每次拿到相同 profile。"""

def random_profile() -> BrowserProfile:
    """随机选择 profile，用于新账号注册。"""
```

**关键约束**：
- `impersonate` 必须与 `user_agent` 中声明的 Chrome 大版本匹配（如 `chrome120` 配 Chrome 120 UA），否则 TLS 指纹与 UA 矛盾反而被识别。
- 所有 Profile 的 `accept_language` 统一为 `en-US,en;q=0.9`，避免重登暴露 `zh-CN`。
- `pick_profile(email)` 用哈希确定性选 profile，保证同一账号重登时拿到与注册时一致的 profile（前提是 device_id 也要持久化，见改动 2）。

### 改动 2：device_id 与 profile 持久化到账号信息

**文件**：`d:\GithubProject\chatgpt2api\services\register\openai_register.py`

**改动点**：
1. `PlatformRegistrar.__init__`（约 386 行）：从 `BrowserProfile` 取指纹特征，替换硬编码的 `user_agent`/`sec_ch_ua`。
2. `worker()` 返回的 `result` dict（597-605 行）新增两个字段：
   ```python

   "fingerprint_profile": self.profile.name,  # profile 名称，便于重登时复用
   ```
3. `register()` 方法中所有用到的 `user_agent`/`sec_ch_ua`/`common_headers`/`navigate_headers` 改为从 `self.profile` 动态构建（当前是模块级全局常量，需改为实例属性或方法）。

**文件**：`d:\GithubProject\chatgpt2api\services\account_service.py`

**改动点**：
1. `_login_with_password`（587 行起）：
   - 从账号信息中读取持久化的 `device_id` 和 `fingerprint_profile`，不再 `str(uuid.uuid4())` 重新生成。
   - 用 `pick_profile(email)` 或保存的 profile name 还原 `BrowserProfile`。
   - `impersonate` 改用 profile 中的值（当前硬编码 `"chrome110"`，600 行）。
   - `accept-language` 改用 profile 中的值（当前硬编码 `zh-CN,zh;q=0.9,en;q=0.8`，641 行）。
   - `sec-ch-ua` 改用 profile 中的值（当前硬编码，643 行）。
2. token 刷新流程（`_OAUTH_USER_AGENT`，36-40 行）同样从 profile 取 UA。

**影响范围**：`add_account_items` 已支持透传任意字段，无需改存储结构。

### 改动 3：sentinel.py 屏幕分辨率与硬件特征从 profile 取

**文件**：`d:\GithubProject\chatgpt2api\utils\sentinel.py`

**改动点**：
1. `SentinelTokenGenerator.__init__`（23 行）新增 `profile: BrowserProfile` 参数（或单独传入 `screen_resolution`、`hardware_concurrency`）。
2. `_get_config()`（41-62 行）：
   - 第 44 行 `"1920x1080"` 改为 `self.profile.screen_resolution`。
   - 第 60 行 `random.choice([4, 8, 12, 16])` 改为 `self.profile.hardware_concurrency`（保持稳定，不随机，否则同账号每次 sentinel 特征不一致）。
3. 第 49 行 sentinel SDK URL：保留硬编码作为默认值，但允许通过参数覆盖，便于 OpenAI 更新 SDK 时快速调整。可选：从 `config.json` 读取 `sentinel_sdk_url`，未配置则用默认值。
4. `build_sentinel_token`（96 行）新增 `profile` 参数透传给 `SentinelTokenGenerator`。

**调用方改动**：`openai_register.py` 中所有 `build_sentinel_token`/`_build_sentinel_token_tuple` 调用传入 `self.profile`。

### 改动 4：注册间隔抖动

**文件**：`d:\GithubProject\chatgpt2api\services\register_service.py`

**改动点**：`_run()`（274-303 行）在 `executor.submit` 前加入随机延迟：
```python
import random as _random
# 提交一个任务后，随机等待 2-6 秒再提交下一个（仅在有空闲线程时）
if self.get()["enabled"] and not self._target_reached(cfg, submitted) and len(futures) < threads:
    submitted += 1
    futures.add(executor.submit(openai_register.worker, submitted))
    # 新增：注册间隔抖动，防同 IP 短时批量注册
    jitter = _random.uniform(2.0, 6.0)
    time.sleep(jitter)
```

**配置化**：在 `register.json` 配置中新增 `register_interval_min` / `register_interval_max`（默认 2.0 / 6.0），允许用户调整。对应 `api/register.py` 的 `RegisterConfigRequest` 新增这两个字段。

### 改动 5：姓名池扩充

**文件**：`d:\GithubProject\chatgpt2api\services\register\openai_register.py`

**改动点**：`_random_name()`（157-160 行）扩充姓名池：
- first_name 从 8 个扩充到 30-50 个（覆盖常见英文名，含部分中性名）。
- last_name 从 7 个扩充到 30-50 个（覆盖常见英文姓氏）。
- 组合数从 56 提升到 900-2500，显著降低重复率。

可选：将姓名池移到独立数据文件 `data/names.json`，便于维护扩展。

### 改动 6（可选）：保活参数微调

**文件**：`d:\GithubProject\chatgpt2api\services\account_service.py`

**改动点**（低优先级，可按需）：
- `_REFRESH_TOKEN_KEEPALIVE_SECONDS`（30 行）从 3 天调整为 2 天（更频繁保活，降低 token 失效概率，但增加请求量，需权衡）。
- `_REFRESH_TOKEN_KEEPALIVE_BATCH_SIZE`（32 行）从 3 调整为 2（更小批量，降低被识别为批量保活的风险）。

此项建议保持默认值，仅在观察到保活失败率上升时调整。

## 四、假设与决策

### 决策 1：指纹 Profile 数量与组合
- 预置 **5 个 Profile**：Chrome 145/Win、Chrome 131/Win、Chrome 131/macOS、Chrome 120/Win、Chrome 120/macOS。
- `curl_cffi` 必须支持对应的 `impersonate` 值（`chrome`、`chrome120`、`chrome131` 等），需在实现前确认 `curl_cffi>=0.15.0` 支持的版本列表。
- **决策依据**：5 个 profile 足以分散指纹聚类，又不至于维护成本过高。

### 决策 2：profile 选择策略
- 新账号注册：`random_profile()` 随机选。
- 重登/刷新：用 `pick_profile(email)` 确定性选（基于 email 哈希），保证与注册时一致。
- **前提**：`device_id` 必须同时持久化（改动 2），否则仅 profile 一致而 device_id 不一致仍会被风控。
- **备选方案**：直接持久化 profile name 到账号信息（改动 2 已包含），重登时读 profile name 而非重新哈希。**采用备选方案**，更可靠。

### 决策 3：注册间隔抖动范围
- 默认 2-6 秒，可通过配置调整。
- 不采用"每注册 N 个账号后长延迟"的策略（复杂度高，且 `check_interval` 已有类似效果）。
- **决策依据**：抖动只需打散并发提交的尖峰，无需过长。

### 决策 4：不改动项
- **不改**邮箱 provider 体系（已足够完善，10 种轮询）。
- **不改** CF clearance / FlareSolverr 机制（已完善）。
- **不改** Outlook 邮箱池状态机（已完善）。
- **不改**密码强度（16 位含特殊字符已足够）。
- **不改** `verify=False`（自签代理场景必要，且改动收益低风险高）。

### 假设
1. `curl_cffi>=0.15.0` 支持 `chrome120`/`chrome131` 等 impersonate 值（实现前需验证）。
2. 账号存储后端（json/sqlite/postgres）的 `add_account_items` 能透传 `device_id`/`fingerprint_profile` 字段（已确认）。
3. OpenAI 不要求 sentinel SDK URL 与实际加载的 SDK 版本严格匹配（当前硬编码已能工作，说明容忍版本差异）。

## 五、验证步骤

### 5.1 单元验证
1. `utils/fingerprint.py`：
   - `pick_profile("test@example.com")` 多次调用返回相同 profile（确定性）。
   - `random_profile()` 返回的 profile 中 `impersonate` 与 `user_agent` 版本一致。
2. `utils/sentinel.py`：传入不同 profile，`_get_config()` 输出的屏幕分辨率、hardwareConcurrency 与 profile 一致。
3. `openai_register.py`：`PlatformRegistrar` 实例的 `self.device_id`、`self.profile` 在注册流程中保持不变。

### 5.2 集成验证
1. 注册一个账号后，检查 `data/accounts.json`（或对应存储）中该账号是否包含 `device_id` 和 `fingerprint_profile` 字段。
2. 触发该账号的重登流程（手动使 refresh_token 失效或调用重登接口），验证：
   - 重登使用的 `device_id` 与注册时一致。
   - 重登使用的 `impersonate`、`accept-language`、`sec-ch-ua` 与注册时一致。
3. 启动批量注册（`threads=3, total=5`），观察日志中注册提交时间戳是否有 2-6 秒间隔。

### 5.3 生存验证（需实际运行）
1. 用改进后的系统注册 10-20 个账号。
2. 观察 24h / 72h / 7 天后的账号存活率，与改进前对比。
3. 重点关注：
   - 注册成功率是否提升（CF/sentinel 失败占比下降）。
   - `app_session_terminated` 是否减少。
   - `account_deactivated` 是否减少。

### 5.4 回归验证
1. 确保改动不破坏现有注册流程（用现有配置跑一轮注册，全流程通过）。
2. 确保改动不破坏现有重登流程（手动触发一个老账号的重登，流程通过）。
3. 确保改动不破坏 token 刷新流程（等待 watcher 触发刷新，流程通过）。
