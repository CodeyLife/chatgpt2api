# 账号限流触发机制分析与修复

## 一、问题现象

用户反馈：free 账号初始额度为 5，额度用完后 quota 显示为 0，但账号 status 仍为"正常"，未触发"限流"，导致账号被继续选中尝试生图。

## 二、限流触发机制概述

限流状态由两个入口设置，二者均依赖 `image_quota_unknown=False`：

### 入口 A：本地扣减（[account_service.py:1324-1332](file:///d:/GithubProject/chatgpt2api/services/account_service.py#L1324-L1332) `mark_image_result`）

```python
if success:
    if not image_quota_unknown:
        next_item["quota"] = max(0, quota - 1)          # 扣减额度
    if not image_quota_unknown and next_item["quota"] == 0:
        next_item["status"] = "限流"                      # 触发限流
    elif next_item.get("status") == "限流":
        next_item["status"] = "正常"                      # ← 限流恢复（BUG 所在）
```

### 入口 B：远程刷新（[openai_backend_api.py:363](file:///d:/GithubProject/chatgpt2api/services/openai_backend_api.py#L363) `get_user_info`）

```python
"status": "正常" if image_quota_unknown and plan_type.lower() != "free"
          else ("限流" if quota == 0 else "正常"),
```

对于 free 账号 + quota=0，无论 `image_quota_unknown` 是 True 还是 False，此表达式结果恒为"限流"——远程刷新路径本身不会漏判。

## 三、根因分析：`mark_image_result` 的 elif 分支错误恢复限流

### 问题代码

[account_service.py:1328-1332](file:///d:/GithubProject/chatgpt2api/services/account_service.py#L1328-L1332)：

```python
if not image_quota_unknown and next_item["quota"] == 0:
    next_item["status"] = "限流"
    next_item["restore_at"] = next_item.get("restore_at") or None
elif next_item.get("status") == "限流":
    next_item["status"] = "正常"   # ← 无条件恢复，未检查 quota
```

`elif` 分支的设计意图：生图成功且额度未耗尽时，将之前误标的"限流"恢复为"正常"。

**缺陷**：当 `image_quota_unknown=True` 且 `quota=0` 时：
- `if not image_quota_unknown and quota == 0` → `False`（因 `image_quota_unknown=True`）
- 落入 `elif status == "限流"` → `True` → **改回"正常"**

但此时 quota=0，根本不应恢复限流。

### 完整触发路径（free 账号）

1. 账号初始：`quota=5, image_quota_unknown=False, status="正常"`
2. 选中生图（预检时 quota=5 通过）
3. 生图过程中，远程刷新（`fetch_remote_info`）发现上游额度已耗尽：
   - 上游 `limits_progress` 不再含 `image_gen` 条目（额度用完后上游可能不再返回该字段）
   - `_extract_quota_and_restore_at`（[openai_backend_api.py:275-280](file:///d:/GithubProject/chatgpt2api/services/openai_backend_api.py#L275-L280)）找不到 `image_gen` → 返回 `(0, None, True)`，即 `quota=0, image_quota_unknown=True`
   - free 账号 → `status="限流"`
   - `update_account` 写入：`quota=0, image_quota_unknown=True, status="限流"`
4. 步骤 2 中已获取 token 的生图请求**成功完成**，调用 `mark_image_result(token, True)`：
   - `current`: `quota=0, image_quota_unknown=True, status="限流"`
   - `success=True`，因 `image_quota_unknown=True` → **不扣减** quota
   - `if not image_quota_unknown and quota == 0` → `False`
   - `elif status == "限流"` → `True` → **status="正常"** ← BUG
   - 写入：`quota=0, image_quota_unknown=True, status="正常"`
5. 结果：账号 `quota=0` 但 `status="正常"`，且因 `image_quota_unknown=True` 使 `_is_image_account_available`（[account_service.py:137-138](file:///d:/GithubProject/chatgpt2api/services/account_service.py#L137-L138)）返回 `True`，账号被继续选中生图，反复失败。

### 关键点

- `_extract_quota_and_restore_at` 把"上游未返回 image_gen 条目"误判为"无限额账号"（`image_quota_unknown=True`），这是触发条件
- `mark_image_result` 的 `elif` 分支无条件恢复限流，未检查 quota 是否真的 > 0，这是直接根因
- 二者叠加导致 free 账号 quota=0 时被错误恢复为"正常"

## 四、修复方案

### 修改文件

[account_service.py:1331-1332](file:///d:/GithubProject/chatgpt2api/services/account_service.py#L1331-L1332)

### 修改内容

```python
# 修改前
elif next_item.get("status") == "限流":
    next_item["status"] = "正常"

# 修改后
elif next_item.get("status") == "限流" and int(next_item.get("quota") or 0) > 0:
    next_item["status"] = "正常"
```

### 修复逻辑验证

| image_quota_unknown | quota | 当前 status | success | 修复前结果 | 修复后结果 |
|---|---|---|---|---|---|
| False | 0 | 正常 | True | 限流 ✓ | 限流 ✓ |
| False | 3 | 限流 | True | 正常 ✓ | 正常 ✓ |
| True | 0 | 限流 | True | 正常 ✗ (BUG) | 限流 ✓ (保持) |
| True | 5 | 限流 | True | 正常 ✓ | 正常 ✓ |
| False | 0 | 限流 | True | 限流 ✓ (if 命中) | 限流 ✓ |

修复后：仅当 `quota > 0` 时才恢复限流为正常，避免 `image_quota_unknown=True` 且 `quota=0` 时错误恢复。

### 副作用评估

- 对真正的无限额账号（`image_quota_unknown=True` 且上游确实不返回 image_gen）：若其 `quota` 字段被远程刷新设为 0 且被误标限流，修复后无法通过 `mark_image_result` 单方面恢复，需依赖下次远程刷新纠正。这是可接受的——限流状态本应由远程刷新权威管理，本地 `mark_image_result` 不应无依据地恢复。
